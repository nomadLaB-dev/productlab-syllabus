## 1. チーム開発のルール：GitHubフロー

一人の時とは違い、チームでは「みんなの共有倉庫（mainブランチ）」を壊さないことが一番大切です。

### 🤝 共同開発の基本
- **`git clone [URL]`**: 仲間が作ったプロジェクトを自分のPCに丸ごとコピーする。
- **`git pull origin main`**: 仲間が更新した最新のコードを自分のPCに取り込む。

### 🛠️ 「合体」させる前にチェックする（PRとレビュー）
プロの現場では、自分のコードをいきなり main には入れません。必ず **プルリクエスト（PR）** を送ります。

1. **ブランチを作る**: 自分の作業専用の「並行世界」を作る。
2. **PRを送る**: 「こんな変更をしました。チェックをお願いします！」と仲間に送る。
3. **レビュー**: 仲間がコードを読み、アドバイスをする。「ここ、もっと良くできるよ！」
4. **マージ（合体）**: OKが出たら、みんなの main に合流！

<Aside type="tip" title="レビューは「プレゼント」🎁">
レビューは間違い探しではなく、**「もっと良いプロダクトにするための相談」**です。自分では気づけないミスを仲間が防いでくれる、最高の安心材料だよ！
</Aside>

## 3. TypeScript：AIに「正しく」伝えるためのルール

「プログラムを書く」とは、AIやコンピュータに指示を出すことです。
でも、**「適当な指示」だと、AIもコンピュータも勘違いしてしまいます。**

### 🚨 JavaScript（自由すぎて怖い）
`「その箱に、なんか入れといて！」`
→ 数字を入れるつもりだったのに、間違えて文字を入れてもコンピュータは気づかず、後でエラーになって爆発します。

### ✨ TypeScript（ルールがあるから安心）
`「その箱には『数字（number）』しか入れちゃダメだよ！」`
→ 間違ったものを入れようとすると、**書いている最中に**「それはルール違反だよ！」と赤線で教えてくれます。

<Aside type="tip" title="AI駆動開発でのメリット">
TypeScriptでしっかり「型（ルール）」を決めると、AIは「あ、この箱は数字なんだな」と正確に理解してくれます。結果として、**AIが生成するコードの精度が爆上がりします！**
</Aside>

## 4. DOM操作：Webサイトを「リモート操作」する

HTML（見た目）をTypeScript（プログラム）から動かす仕組みを **DOM（ドム）** と呼びます。

### 🕹️ リモコンで操作するイメージ
1. **狙いを定める**: `document.getElementById('target')` 「あのIDのボタンを掴んで！」
2. **命令する**: `.textContent = "変更！"` 「文字を書き換えて！」

### 📦 「中身を教える」という魔法（型アサーション）
TypeScriptはHTMLの中身を覗けません。だから、プログラムからHTMLを触るときは、私たちが「これは何ボタンだよ」と教えてあげる必要があります。

```typescript
// 1. HTMLから「送信ボタン」を探す。
// 2. でもTSはそれが「ボタン」か確信が持てない...
// 3. だから「これはボタン（HTMLButtonElement）だよ！」と教える（as を使う）
const btn = document.getElementById('submit-btn') as HTMLButtonElement;

btn.addEventListener('click', () => {
  alert('クリックされたよ！');
});
```

## 5. Next.jsへの第一歩：JSX（HTML風の書き方）

これからやるモダンな開発では、.html ファイルにちまちま書くことはしません。 TypeScriptの中に、直接HTMLっぽい見た目を書く JSX という技術を使います。

 - 今まで: HTMLで骨組みを作って、後からTSでリモコン操作。
 - これから: TSの中で「どんな見た目にするか」を最初から宣言する。

<Aside type="note"> 
「データが変わったら、見た目も勝手に変わる」という魔法のような仕組み（React）を動かすための準備運動だと思ってね！ 
</Aside>
